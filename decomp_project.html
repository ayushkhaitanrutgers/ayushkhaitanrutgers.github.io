LLM-CAS: A Decomposition-Driven Framework for Asymptotic Inequalities

A formal workflow that couples frontier language models with a computer algebra system to propose and verify asymptotic inequalities over appropriately chosen subdomains.

This article describes the Decompositions Project, an implementation of an LLM-CAS framework that aims to verify asymptotic inequalities by combining two capabilities: (i) a large language model (LLM) that proposes natural domain decompositions, and (ii) a computer algebra system (CAS) that attempts a rigorous verification of each decomposed region. In building this project, we confirm Terry Tao's prediction that an AI tool can help rigorously prove asymptotic inequalities. Our tool is able to completely prove all questions asked by Terry here.

Overview

The Decompositions Project implements a practical loop between an LLM and a CAS. The LLM suggests a small set of subdomains where the comparison is straightforward; the CAS (Mathematica) then attempts a formal verification on each subdomain using first-order methods. The repository is available at:

https://github.com/ayushkhaitanrutgers/decompositions-project

Definition

Let \(f\) and \(g\) be non-negative functions on a domain \(D\). We write \(f \ll g\) if there exists a constant \(C > 0\) such that

\(\forall x \in D,\; f(x) \le C \cdot g(x).\)

This is the usual big-O relation, stated uniformly on \(D\).

What the Tool Proves

The framework currently verifies two representative results. The statements below are presented in a concise, shareable form.

1) Series bound

For parameters \(h > 1\), \(m > 1\), consider the series

\[ S(h,m) \;:=\; \sum_{d=0}^{\infty} \frac{2d+1} {2h^2\left(1+\frac{d(d+1)}{h^2}\right) \left(1+\frac{d(d+1)}{h^2 m^2}\right)^2}. \]

The tool verifies the asymptotic inequality

\[ S(h,m) \;\ll\; 1 + \log\!\big(m^2\big) \quad \text{uniformly for } h > 1,\; m > 1. \]

In practice, the LLM proposes breakpoints in the index \(d\) (where the denominator’s behavior changes), and Mathematica confirms the bound on each range using symbolic inequalities.

2) Functional inequality

For variables \(x,y\) with \(x > 0\) and \(y > 1\), the framework verifies

\[ x\,y \;\le\; y \,\log y \;+\; e^{x}. \]

The verification proceeds by splitting the domain into regimes where comparison is direct (for example, ranges where one side clearly dominates) and applying \(\texttt{Resolve}\) to certify each regime.

How It Works

  1. Problem setup. Specify a functional inequality \(f,g,D\) or a series \(S\) with a proposed upper bound \(g\) (see examples.py in the repository).
  2. Decomposition. The LLM proposes subdomains (for functions) or index breakpoints (for series) that aim to simplify each local comparison.
  3. Simplification. Where needed, the pipeline replaces complicated summands with valid dominant forms on each subdomain (e.g., using leading terms under positivity conditions).
  4. Verification. Mathematica’s Resolve attempts a first-order proof that \(f \le C \cdot g\) (or the corresponding series bound) on each piece, searching over a finite grid of constants \(C\).
  5. Conclusion. If every piece is certified, the global inequality follows.

Why Mathematica?

Off-the-shelf SMT solvers and first-order provers often struggle with transcendental functions. Mathematica’s Resolve has been effective for such cases and returns True only when a symbolic proof is found within its first-order scope. While Mathematica does not emit a proof object, this approach has been reliable for the problems targeted here.

Relation to Notes by Terence Tao

The project follows two ideas: (a) propose helpful decompositions and (b) verify each piece symbolically. For context, see Tao’s blog post “A proof of concept tool to verify estimates” and his MathOverflow remarks referenced in the repository, including this answer.

Repository and Usage

  • Repository: Decompositions Project
  • Dependencies: Python 3.9+, access to an LLM API (e.g., Gemini), and Mathematica with wolframscript on PATH.
  • Environment: place your API key in .env (e.g., GOOGLE_API_KEY=...), set WOLFRAMSCRIPT if needed.

Install

pip install -r requirements.txt
echo "GOOGLE_API_KEY=your_api_key_here" > .env    # or GEMINI_API_KEY
# If not auto-detected:
export WOLFRAMSCRIPT=/usr/local/bin/wolframscript

Run

Sanity-check the CAS bridge:

python mathematica_export.py

Try a functional inequality you added to examples.py:

decomp prove question_<id>

Try a series inequality you added to examples.py:

decomp series series_<id>

Notes and Limitations

  • Proof artifacts. Mathematica does not output a checkable proof object; integrating with a proof assistant is a natural extension.
  • Transcendental scope. Some mixed symbolic–transcendental statements may require further decomposition or tighter dominant bounds.
  • Choosing constants. The constant search is finite; widen it or refine the decomposition if a piece fails.
  • Upper bounds from LLMs. Asking the LLM to guess tight summand bounds is still challenging; regime-wise CAS simplification has been more reliable.
